# ZK-AuthaaS Simulation

A distributed zero-knowledge proof verification system built to study load-balancing behaviour across heterogeneous verifier pools. The system is composed of a FastAPI request handler, a custom verifier selector that routes proofs by scheme and weighted cost, and pools of SNARK and STARK verifier workers connected through Redis queues. Load is driven by k6.

SNARK verification uses **real Groth16 proof verification** via `snarkjs` running inside each worker container. STARK verification uses a timing placeholder. Load is generated by k6 — either from a local machine for development, or from a dedicated k6 EC2 instance in the same AWS VPC for accurate cloud benchmarking.

This project is deployed with **Docker Swarm** so that verifier counts can be scaled with a single command from 10 up to 1000 replicas without editing the compose file.

## 1. Prerequisites

### For local development (your laptop)

- [Docker Desktop](https://www.docker.com/products/docker-desktop/) (Windows / macOS) or Docker Engine (Linux) — Swarm mode is included
- [k6](https://k6.io/docs/getting-started/installation/) for load generation
- [Python 3.8+](https://www.python.org/) for the sweep and visualization scripts
- Python packages: `pip install pandas matplotlib numpy`

> **Note:** Node.js is **not** required on your local machine. The SNARK worker runs inside a `node:18-slim` Docker container — the build handles everything.

On Windows PowerShell the monitoring script also requires loosening the execution policy once:
```powershell
Set-ExecutionPolicy -Scope CurrentUser RemoteSigned
```

### For AWS deployment

Two EC2 instances are required (see [Section 10](#10-deploying-on-aws)):

| Instance | Role | Suggested type |
|---|---|---|
| `zk-authaas-backend` | Runs the Docker Swarm stack | `c5.4xlarge` (50+50 verifiers) |
| `zk-authaas-k6` | Runs k6 load generator | `t3.small` |

Both instances must be in the **same VPC and Availability Zone**. k6 fires at the backend over the private network, keeping network overhead below 1 ms and out of the measurements.

## 2. Architecture

```
 ┌─────────┐   POST /verify/submit   ┌─────────────────┐
 │   k6    │ ──────────────────────► │ request-handler │
 └─────────┘                         │   (FastAPI)     │
      ▲                              └────────┬────────┘
      │ poll /verify/status                   │ lpush proof_queue
      │                                       ▼
      │                              ┌─────────────────┐
      │                              │   proof-queue   │  (Redis :6379)
      │                              └────────┬────────┘
      │                                       │ brpop proof_queue
      │                                       ▼
      │                              ┌─────────────────┐
      │                              │ verifier-       │  cost-weighted least-queue
      │                              │   selector      │  + pub/sub feedback
      │                              └────────┬────────┘
      │                         lpush snark_queue:{i}   lpush stark_queue:{i}
      │                                       ▼                     ▼
      │                              ┌───────────────┐   ┌───────────────┐
      │                              │  snark-queue  │   │  stark-queue  │
      │                              │ (Redis :6380) │   │ (Redis :6381) │
      │                              └───────┬───────┘   └───────┬───────┘
      │          brpop snark_queue:{i}       │                   │  brpop stark_queue:{i}
      │                                      ▼                   ▼
      │                              ┌──────────────┐   ┌──────────────┐
      │                              │snark-verifier│   │stark-verifier│
      │                              │ (N replicas) │   │ (N replicas) │
      │                              │  snarkjs     │   │  placeholder │
      │                              │ Groth16 real │   │  timing only │
      │                              └──────┬───────┘   └──────┬───────┘
      │                                     │                  │
      │                                     └────── publish ───┴── verifier_feedback
      │                                                        │
      └──────── set status:{job_id} ───────────────────────────┘
```

A single Redis per verifier type hosts one list per worker node (`snark_queue:{index}` / `stark_queue:{index}`). Each worker reads only its own key, giving the selector fine-grained routing control while keeping the total number of Redis processes small (3 regardless of replica count).

Each SNARK worker loads `verification_key.json` once at startup and calls `snarkjs.groth16.verify(vKey, publicSignals, proof)` on every job. The measured latency therefore reflects real BN128 pairing computation, not a sleep.

## 6. Step 1 — Find True Capacity (VU Sweep)

Run the VU sweep with **weight=0** (pure least-queue, cost ignored). This removes the weight as a variable so the knee you find is the system's true throughput ceiling — independent of any routing decision. The knee VU recorded here is used as the fixed load in Step 2 and as the centre of the VU range in Step 3.

### Set the selector to weight=0

Use the `/admin/set-weight` endpoint — no Docker access needed, works from any machine that can reach port 8000:

**Local:**
```bash
curl -X POST "http://localhost:8000/admin/set-weight?snark=0&stark=0"
```

**AWS (run from k6 EC2 or your laptop):**
```bash
curl -X POST "http://<backend-private-ip>:8000/admin/set-weight?snark=0&stark=0"
```

Confirm the selector received the change:
```bash
curl -s "http://localhost:8000/admin/get-weight"
# Expected: {"snark_cost_weight": 0.0, "stark_cost_weight": 0.0}
```

### Run the sweep

**Local (10 + 10 verifiers):**
```bash
python sweep_throughput.py \
  --vus 10,20,30,50,75,100,150 \
  --iterations-per-vu 10 \
  --cooldown 10 \
  --stark-ratio 0.0 \
  --output sweep_baseline.csv \
  --clean
```

**AWS (50 + 50 verifiers, run on k6 EC2):**
```bash
python3 sweep_throughput.py \
  --target <backend-private-ip> \
  --vus 25,50,100,150,200,300,400 \
  --iterations-per-vu 10 \
  --cooldown 15 \
  --stark-ratio 0.0 \
  --output sweep_baseline.csv \
  --clean
```

Plot and read the knee:
```bash
python visualize_sweep.py --input sweep_baseline.csv --output sweep_baseline_graph.png
```

**What to record:** the VU level just before `throughput_req_per_sec` flattens — write it down as **`KNEE_VU`**. This value is used as the fixed `--vus` in Step 2 and as the centre of the VU range in Step 3.

## 7. Step 2 — Find Optimal Cost-Weight (Weight Sweep)

The `--snark-cost-weight` parameter controls how strongly routing penalises expensive nodes:

```
score(node i) = queue_depth(i) + cost(i) × snark_cost_weight
```

| Weight value | Routing behaviour |
|---|---|
| `0` | Pure least-queue — ignores node cost entirely (used in Step 1) |
| `10` | Default — balances queue depth and cost |
| Very high | Always routes to cheapest node regardless of queue depth |

`weight_sweep.py` steps through weight values, calls `POST /admin/set-weight` at each iteration (the selector reads the new value on its very next job dispatch — no container restart), runs k6 at the fixed `KNEE_VU` load from Step 1, and records throughput plus average cost per job. `visualize_weight_sweep.py` then plots a **composite score** (`throughput / avg_cost`) that peaks at the optimal weight.

### Local sweep (laptop)

```bash
python weight_sweep.py \
  --weights 0,1,2,3,5,7,10,15,20,30,50 \
  --vus <KNEE_VU> \
  --iterations 1000
```

Then plot:
```bash
python visualize_weight_sweep.py   # produces weight_sweep_graph.png
```

The graph has two panels:
1. **Top** — throughput (req/s) and p95 latency vs weight; marks peak throughput
2. **Bottom** — average cost per job and composite score (throughput / cost); marks the optimal weight

### AWS sweep (k6 EC2)

The sweep now uses the HTTP API to change weights, so it no longer needs Docker access. It runs on the **k6 EC2** alongside the other sweep scripts — no changes to the backend EC2 are required.

Copy `weight_sweep.py` to the k6 EC2 (run on your laptop):

**Git Bash:**
```bash
scp -i "zk-authaas-key.pem" weight_sweep.py ubuntu@<k6-public-ip>:~/
```

**PowerShell:**
```powershell
scp -i "zk-authaas-key.pem" weight_sweep.py ubuntu@<k6-public-ip>:~/
```

Run the sweep on the k6 EC2:
```bash
python3 weight_sweep.py \
  --target <backend-private-ip> \
  --weights 0,1,2,3,5,7,10,15,20,30,50 \
  --vus <KNEE_VU> \
  --iterations 2000
```

Copy results back and visualize:
```bash
# Git Bash on your laptop:
scp -i "zk-authaas-key.pem" ubuntu@<k6-public-ip>:~/weight_sweep_results.csv .

```

**What to record:** the weight where the composite score peaks — write it down as **`BEST_WEIGHT`**.

**Set `BEST_WEIGHT` for the comparison sweep.** Use `POST /admin/set-weight` for a session-level change (takes effect immediately, resets to CLI default on container restart):

```bash
curl -X POST "http://<backend-private-ip>:8000/admin/set-weight?snark=<BEST_WEIGHT>&stark=1.0"
curl -s "http://<backend-private-ip>:8000/admin/get-weight"
# Expected: {"snark_cost_weight": <BEST_WEIGHT>, "stark_cost_weight": 1.0}
```

Or use `docker service update` on the backend EC2 to make it **persistent** (survives container restarts):
```bash

    --proof-host proof-queue --proof-port 6379 \

    --stark-host stark-queue --stark-port 6379 \
```

## 8. Step 3 — Compare Routing Algorithms

This is the main experiment: weighted least-queue at `BEST_WEIGHT` vs plain round-robin. The comparison measures both **performance** (throughput and latency) and **cost efficiency** (average cost per dispatched job).

> **Before starting:** confirm the selector is locked to weighted + `BEST_WEIGHT` (the lock command at the end of Step 2). All sweep commands use the same VU range as Step 1 so the graphs share the same x-axis scale.

### Cost model

Nodes alternate between cost `1.0` (cheap) and cost `2.0` (expensive). With 10 default nodes this gives 5 cheap + 5 expensive.

| Algorithm | Routing decision | Expected avg cost per job |
|---|---|---|
| Weighted least-queue | Fills cheap nodes first; spills to expensive only when saturated | ≈ 1.0 at low load, rising toward 1.5 as load increases |
| Round-robin | Distributes evenly regardless of cost | Always ≈ 1.5 |

The gap between the two lines in the cost panel is the key result.

### Step 1 — Confirm selector is weighted at BEST_WEIGHT

```bash
docker service logs zk_verifier-selector --tail 1
# Must show: Routing=weighted, SNARK_COST_WEIGHT=<BEST_WEIGHT>
```

### Step 2 — Run the weighted sweep

**Local (10 + 10):**
```bash
python sweep_throughput.py \
  --vus 10,20,30,50,75,100,150 \
  --iterations-per-vu 10 \
  --cooldown 10 \
  --stark-ratio 0.0 \
  --output sweep_weighted.csv \
  --clean
```

**AWS (50 + 50, run on k6 EC2):**
```bash
python3 sweep_throughput.py \
  --target <backend-private-ip> \
  --vus 25,50,100,150,200,300,400 \
  --iterations-per-vu 10 \
  --cooldown 15 \
  --stark-ratio 0.0 \
  --output sweep_weighted.csv \
  --clean
```

### Step 3 — Switch the selector to round-robin


Wait ~10 seconds, then confirm:
```bash

```

### Step 4 — Run the round-robin sweep

**Local:**
```bash

  --vus 10,20,30,50,75,100,150 \
  --iterations-per-vu 10 \
  --cooldown 10 \
  --stark-ratio 0.0 \

```

**AWS (k6 EC2):**

This produces `comparison_graph.png` with panels:
1. **Throughput** — req/s vs VUs, both algorithms, peak annotated
2. **Latency** — p95 and p99 ms vs VUs
3. **Avg cost per job** — the key result; shaded region shows cost saved by weighted routing vs round-robin, with reference lines at `1.0` (theoretical best) and `1.5` (round-robin theory)

## 9. Step 4 — Run the Load Test and Visualize

Run one final, long k6 test using the `KNEE_VU` and `BEST_WEIGHT` values determined in Steps 1 and 2. This produces a time-series CSV (`test_results.csv`) that captures throughput and latency over the entire run duration as a single continuous graph.

### Run the test

**Local (laptop → localhost):**
```bash
k6 run \
  -e VUS=<KNEE_VU> \
  -e ITERATIONS=<KNEE_VU * 20> \
  -e STARK_RATIO=0.0 \
  load_test.js \
  --out csv=test_results.csv
```

**AWS (k6 EC2 → backend private IP):**
```bash
k6 run \
  -e TARGET=<backend-private-ip> \
  -e VUS=<KNEE_VU> \
  -e ITERATIONS=<KNEE_VU * 20> \
  -e STARK_RATIO=0.0 \
  load_test.js \
  --out csv=test_results.csv
```

Always use the **private IP** of the backend EC2, not the public IP. Private IP traffic stays on the AWS internal network and adds less than 1 ms RTT, keeping it out of the `async_verification_time` measurements.

Copy the CSV back to your laptop (AWS only):
```bash
# Git Bash on your laptop:
scp -i "zk-authaas-key.pem" ubuntu@<k6-public-ip>:~/test_results.csv .
```

### Visualize

```bash
python visualize_k6.py   # produces k6_performance_graph.png
```

### Configuration reference

All load test parameters are environment variables — no need to edit the script:

```bash
k6 run -e TARGET=localhost   load_test.js   # local (default)
k6 run -e STARK_RATIO=0.0    load_test.js   # pure SNARK
k6 run -e STARK_RATIO=1.0    load_test.js   # pure STARK
k6 run -e STARK_RATIO=0.5    load_test.js   # 50/50 mix
```

### Suggested load profiles

| Deployment | Verifiers | KNEE_VU (typical) | ITERATIONS |
|---|---|---|---|
| Local laptop | 10 + 10 | 30 – 75 | `KNEE_VU × 20` |
| AWS `c5.4xlarge` | 50 + 50 | 100 – 300 | `KNEE_VU × 20` |
| AWS `c5.24xlarge` | 500 + 500 | 500 – 1 000 | `KNEE_VU × 20` |

The `KNEE_VU × 20` formula gives roughly the same wall-clock run time regardless of VU count, which keeps the time-series graph a consistent length across deployments.

## 11. Deploying on AWS

### Account access

This project uses a **paid-tier AWS account** (borrowed from a professor). Do not use AWS Educate or free-tier accounts — service limits and network restrictions on those tiers interfere with Docker Swarm overlay networking and spot instance availability.

Contact the account holder before starting any session and confirm the current billing balance. Set a **budget alert at $20** in AWS Billing before launching anything.

### Two-instance layout

```
Your laptop  ──SSH──►  zk-authaas-backend (c5.4xlarge)   ← Docker Swarm stack
                               │  AWS private network (same VPC + AZ)
             ──SSH──►  zk-authaas-k6 (t3.small)  ──HTTP:8000──►  backend private IP
```

### Step 1 — IAM and SSH (one-time per team)

- Log in as an IAM user with `AmazonEC2FullAccess` (do not use the root account)
- Go to **EC2 → Key Pairs → Create key pair**, name it `zk-authaas-key`, download the `.pem` file
- `chmod 400 zk-authaas-key.pem`
- Pick one region and use it for every resource in this project (`us-east-1` is a good default)

### Step 2 — Launch the backend EC2

1. **EC2 → Launch Instance**
2. Name: `zk-authaas-backend`
3. AMI: **Ubuntu 22.04 LTS**
4. Instance type: `c5.4xlarge` (16 vCPU, 32 GB RAM)
5. Key pair: `zk-authaas-key`
6. Security group — create `zk-authaas-backend-sg` with:
   - SSH (port 22) from **My IP**
   - Custom TCP (port 8000) from **My IP** *(for manual curl checks)*
7. Storage: 30 GB gp3
8. Launch — record the **Public IPv4** and the **Private IPv4**

### Step 3 — Launch the k6 EC2

1. **EC2 → Launch Instance**
2. Name: `zk-authaas-k6`
3. AMI: **Ubuntu 22.04 LTS**
4. Instance type: `t3.small` (2 vCPU, 2 GB RAM)
5. Key pair: `zk-authaas-key`
6. Network — **critical**: expand and set:
   - VPC: **same VPC** as the backend
   - Subnet: **same subnet / AZ** as the backend (check the backend's AZ in the EC2 console and match it exactly)
7. Security group — create `zk-authaas-k6-sg`:
   - SSH (port 22) from **My IP**
   - *(No inbound rules needed — k6 only makes outbound calls)*
8. Storage: 8 GB gp3
9. Launch — record the **Private IPv4**

### Step 4 — Allow k6 to reach the backend

1. **EC2 → Security Groups → `zk-authaas-backend-sg` → Inbound rules → Edit**
2. Add rule: Custom TCP · Port 8000 · Source: `<k6-private-ip>/32`
3. Save

### Step 5 — Set up the backend EC2


## 13. Troubleshooting

**Services stuck in `0/N` or restarting.**
Check the failure reason: `docker service ps zk_snark-verifier --no-trunc`. Most common causes are memory overcommit (reduce replicas or raise the memory limit in `docker-compose.yml`) or a failed image build.

**SNARK workers show `Error: Invalid proof` in logs.**
The `verification_key.json` inside the image does not match the proof being sent. Confirm the file was copied to the project root before running `docker compose build`, then rebuild with `docker compose build --no-cache zk_snark-verifier`.

**SNARK workers are slower than expected / high CPU.**
Normal — `snarkjs.groth16.verify()` performs BN128 elliptic curve pairings which are genuinely compute-intensive. If replicas are saturated, scale up: `docker service scale zk_snark-verifier=50`.

**k6 reports many `submit_failures`.**
The request-handler is saturated. Scale it: `docker service scale zk_request-handler=3`.

**k6 reports correct job submissions but `failed_verifications` are high.**
The SNARK workers are processing jobs but returning `false`. This usually means the proof or public signals in `load_test.js` were modified and no longer match the verification key. Restore `SNARK_PROOF` and `SNARK_PUBLIC_SIGNALS` from `zk-authaas-snark/Test/`.

**Queues stay full and never drain.**
Either no worker replicas are running (`docker service ls`) or the selector's `--snark-count` / `--stark-count` don't match the actual replica count, so some queue keys receive jobs that no worker is listening on.

**k6 EC2 gets `connection refused` on port 8000.**
Either the backend security group is missing the rule for the k6 EC2's private IP (Section 9 Step 4), or you used the backend's public IP instead of the private IP in the `TARGET` variable.

**Cost panel in `comparison_graph.png` shows `None` or is missing.**
The stack was deployed before the `/stats/cost` endpoint existed. Rebuild and redeploy the request-handler image: `docker compose build --no-cache` then `docker stack deploy -c docker-compose.yml zk`.

**Cost panel shows the same value for both algorithms (both ≈ 1.5).**
The selector was not actually switched between runs — check the selector log (`docker service logs zk_verifier-selector --tail 3`) before each sweep to confirm `Routing=weighted` or `Routing=roundrobin`.

**Weight sweep graph shows flat cost (≈ 1.5) at all weights.**
The stack may not have the `/stats/cost` endpoint or the selector was started without cost-weight args. Rebuild and redeploy: `docker compose build --no-cache && docker stack deploy -c docker-compose.yml zk`. Then confirm the selector log shows `SNARK_COST_WEIGHT=<value>`.

**Weight sweep finishes but throughput barely changes across weights.**
The VU count is too low to saturate any node, so the routing decision makes no difference. Increase `--vus` until at least some nodes are non-empty (`monitor_queues.ps1` shows queue depth > 0).

**`docker service update` in weight sweep fails with "service not found".**
The Docker service name defaults to `zk_verifier-selector`. If you deployed under a different stack name, pass `--service <your-stack>_verifier-selector`.

**`docker stack rm` leaves networks behind.**
Wait ~15 seconds before redeploying. Swarm tears down overlay networks asynchronously.

**`command not found: watch` on Windows.**
That is a Linux command. Use `monitor_queues.ps1` instead.
