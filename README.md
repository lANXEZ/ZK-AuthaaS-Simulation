
## ZK-AuthaaS Simulation: Project Guide

This project simulates a scalable zero-knowledge proof verification system using Docker Compose, Python, and k6 for load testing. It features a modular architecture with separate Redis queues and workers for SNARK and STARK proofs, a FastAPI request handler, and a verifier selector for load distribution.

---

### 1. Prerequisites

- [Docker Desktop](https://www.docker.com/products/docker-desktop/) (required)
- [Node.js](https://nodejs.org/) (for k6 load testing)
- [k6](https://k6.io/) (install via `npm install -g k6` or from their website)
- [Python 3.x](https://www.python.org/) (for result visualization)
- Python packages: `pandas`, `matplotlib`, `numpy` (install via `pip install pandas matplotlib numpy`)

---

### 2. Build and Start All Services

Open a terminal in this project directory and run:

```
docker-compose up --build
```

This will start the following containers:
- `proof-queue`: Redis instance for incoming proof requests
- `snark-queue-N` and `stark-queue-N`: Redis instances for SNARK and STARK verifiers (N = 1, 2, ...)
- `request-handler`: FastAPI service (port 8000) for `/verify` endpoint
- `verifier-selector`: Python service that routes proofs to available verifiers
- `snark-verifier-N` and `stark-verifier-N`: Worker containers for SNARK and STARK proof verification

**Scaling:**
- The number of SNARK and STARK verifiers is controlled by duplicating the relevant service blocks in `docker-compose.yml` and adjusting the `--snark-count` and `--stark-count` arguments for the selector.
- Each verifier has its own queue and runs as a separate container.

---

### 3. Run the Load Test

To avoid connection errors, wait for the API to be ready before starting the load test.

1. Start all services:
	```
	docker-compose up --build
	```
2. In a new terminal, run:
	```
	k6 run load_test.js
	```
	This will send a high volume of requests to `http://localhost:8000/verify` using the configuration in `load_test.js`.

**Note:** You can edit `load_test.js` to change request patterns, payloads, or load profile.

---

### 4. Monitor the Queues

You can check the length of each queue to observe system load and processing:

**Snapshot (one-time):**
```
docker exec -it proof-queue redis-cli LLEN proof_queue
docker exec -it snark-queue-1 redis-cli LLEN snark_queue
docker exec -it stark-queue-1 redis-cli LLEN stark_queue
```

**Real-time (repeat every second):**
```
docker exec -it proof-queue redis-cli -r -1 -i 1 LLEN proof_queue
docker exec -it snark-queue-1 redis-cli -r -1 -i 1 LLEN snark_queue
docker exec -it snark-queue-2 redis-cli -r -1 -i 1 LLEN snark_queue
... (repeat for all snark-queue-N)
docker exec -it stark-queue-1 redis-cli -r -1 -i 1 LLEN stark_queue
docker exec -it stark-queue-2 redis-cli -r -1 -i 1 LLEN stark_queue
... (repeat for all stark-queue-N)
```

---

### 5. Visualize Performance Results

After running the load test, k6 can output results as a CSV (see k6 docs for CSV output). To visualize throughput and latency:

1. Ensure `test_results.csv` is present (generated by k6 or provided).
2. Run the visualization script:
	```
	python visualize_k6.py
	```
3. This will generate `k6_performance_graph.png` with RPS and latency metrics.

---

### 6. Stopping the System

To stop all containers:
```
docker-compose down
```

---

### 7. Notes

- All core logic is in Python (`requestHandler.py`, `verifierSelector.py`, `SNARKVerifierWorker.py`, `STARKVerifierWorker.py`).
- The load test is in `load_test.js` (edit as needed).
- Visualization is handled by `visualize_k6.py` (requires Python and dependencies).
- For troubleshooting, check logs with:
  ```
  docker-compose logs -f
  ```
