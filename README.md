# ZK-AuthaaS Simulation

A distributed zero-knowledge proof verification system that demonstrates **cost-aware load balancing** across heterogeneous verifier pools. The system routes incoming proofs through a custom selector that combines least-queue scheduling with a per-node cost weight, then dispatches them to a pool of verifier workers via Redis queues. Load is generated by k6.

SNARK verification uses **real Groth16 proof verification** via `snarkjs` running inside each worker container, with the BN128 pairing actually computed on every request. STARK verification uses a calibrated timing placeholder.

The canonical deployment is a **two-node Docker Swarm cluster** with 80 SNARK + 80 STARK verifier workers and 8 request-handler replicas, sustaining ~1,800 req/s of real Groth16 verification throughput on AWS `c5.24xlarge` hardware.

## 1. Final Architecture

```
                      ┌──────────────────────┐
   your laptop ──┐    │  Manager EC2         │
                 │    │  (c5.24xlarge)       │
                 │    │  • Redis × 3         │
                 │    │  • request-handler×8 │
                 │    │  • verifier-selector │
                 ├───►│  • STARK pool × 80   │
                 │    └─────────┬────────────┘
                 │              │ Swarm overlay (VXLAN)
                 │              ▼
                 │    ┌──────────────────────┐
                 │    │  Worker EC2          │
                 │    │  (c5.24xlarge)       │
                 │    │  • SNARK pool × 80   │
                 │    └──────────────────────┘
                 │
                 │    ┌──────────────────────┐
                 └───►│  k6 EC2 (c5.2xlarge) │
                      │  • Load generator    │
                      └──────────┬───────────┘
                                 │ HTTP :8000
                                 ▼
                          (manager EC2)
```

| Role | Instance | Purpose |
|---|---|---|
| Manager | `c5.24xlarge` (96 vCPU / 192 GiB) | Redis ×3, request-handler ×8, verifier-selector, STARK pool (80 workers × 0.15 vCPU) |
| Worker | `c5.24xlarge` (96 vCPU / 192 GiB) | SNARK pool only (80 workers × 1.0 vCPU each) |
| k6 | `c5.2xlarge` (8 vCPU / 16 GiB) | Load generator (k6 + Python sweep scripts) |

**Why two backend nodes:** a single c5.24xlarge cannot host 80 SNARK workers at 1.0 vCPU plus the rest of the stack (it runs out of vCPU), and overlay-network VIPs become unstable above ~1000 containers per node. Splitting SNARK onto a dedicated worker node solves both. See [Section 9](#9-known-architectural-limitations) for the full reasoning.

### Data path

```
 ┌─────────┐   POST /verify/submit   ┌─────────────────┐
 │   k6    │ ──────────────────────► │ request-handler │ × 8 replicas
 └─────────┘                         │   (FastAPI)     │ Swarm ingress mesh
      ▲                              └────────┬────────┘ load-balances
      │ poll /verify/status                   │ lpush proof_queue
      │                                       ▼
      │                              ┌─────────────────┐
      │                              │   proof-queue   │  (Redis :6379)
      │                              └────────┬────────┘
      │                                       │ brpop proof_queue
      │                                       ▼
      │                              ┌─────────────────┐
      │                              │ verifier-       │  cost-weighted least-queue
      │                              │   selector      │  + pub/sub feedback
      │                              │      (v2)       │  cached weights, pipelined ops
      │                              └────────┬────────┘
      │                         lpush snark_queue:{i}   lpush stark_queue:{i}
      │                                       ▼                     ▼
      │                              ┌───────────────┐   ┌───────────────┐
      │                              │  snark-queue  │   │  stark-queue  │
      │                              │ (Redis :6380) │   │ (Redis :6381) │
      │                              └───────┬───────┘   └───────┬───────┘
      │          brpop snark_queue:{i}       │                   │  brpop stark_queue:{i}
      │                                      ▼                   ▼
      │                              ┌──────────────┐   ┌──────────────┐
      │                              │snark-verifier│   │stark-verifier│
      │                              │ × 80 replicas│   │ × 80 replicas│
      │                              │  snarkjs     │   │  placeholder │
      │                              │ Groth16 real │   │  timing only │
      │                              └──────┬───────┘   └──────┬───────┘
      │                                     │                  │
      │                                     └────── publish ───┴── verifier_feedback
      │                                                        │
      └──────── set status:{job_id} ───────────────────────────┘
```

A single Redis per verifier type hosts one list per worker (`snark_queue:{index}` / `stark_queue:{index}`). Each worker reads only its own key, giving the selector fine-grained routing control while keeping the total number of Redis processes small (3 regardless of replica count).

Each SNARK worker loads `verification_key.json` once at startup and calls `snarkjs.groth16.verify(vKey, publicSignals, proof)` on every job. The measured latency therefore reflects real BN128 pairing computation, not a sleep.

### Cost-weighted routing

Verifier nodes alternate between cost `1.0` (cheap) and cost `2.0` (expensive). With 80 SNARK nodes this gives 40 cheap + 40 expensive. The selector chooses worker `i` for each job using:

```
score(i) = pseudo_queue[i] + cost[i] × snark_cost_weight
worker = argmin score
```

Pseudo-queue depth is incremented on dispatch and decremented when a worker publishes completion via Redis pub/sub. The weight is live-adjustable via `POST /admin/set-weight` without restarting the selector — the v2 selector caches it in memory and refreshes from Redis every 200 ms via a background thread.

| weight | Behaviour |
|---|---|
| `0` | Pure least-queue — cost ignored. Used to find raw capacity. |
| `1.0` (default) | Light cost preference — cheap nodes filled first, spillover only when saturated |
| Higher values | Stronger cost preference — creates a deeper "moat" before any expensive node is used |

| Algorithm | Routing decision | Avg cost per job |
|---|---|---|
| Weighted least-queue | Cheap nodes filled first; spills to expensive only when cheap is saturated | ≈ 1.0 below saturation, rises toward 1.5 under overload |
| Round-robin | Distributes evenly regardless of cost | Always ≈ 1.5 |

The gap between these two cost lines is the system's headline finding: **33% cost reduction at equivalent throughput** under non-overload conditions.

## 3. Local Quickstart (single-node)

For development and debugging on a laptop. The same compose file deploys a smaller-scale single-node version.

```bash
docker swarm init
docker compose build                           # ~3–5 min first run (snarkjs install)
docker stack deploy -c docker-compose.yml zk
docker service ls                              # wait until all replicas reach target
```

The default deployment is **80+80 verifiers**, which a laptop usually can't run. Scale down for local work:
```bash
docker service scale zk_snark-verifier=10 zk_stark-verifier=10
docker service update \
  --args "python verifierSelector.py --proof-host proof-queue --proof-port 6379 --snark-host snark-queue --snark-port 6379 --stark-host stark-queue --stark-port 6379 --snark-count 10 --stark-count 10 --routing weighted --snark-cost-weight 1.0 --stark-cost-weight 1.0" \
  zk_verifier-selector
```

Smoke test:
```bash
k6 run load_test.js
# Expected: failed_verifications=0, submit_failures=0
```

## 4. Monitor the Queues Live

Watch all queues in real time with colour-coded depth:
```powershell
.\monitor_queues.ps1
.\monitor_queues.ps1 -SnarkCount 80 -StarkCount 80 -Refresh 2
```

It displays the incoming `proof_queue` and every `snark_queue:{i}` / `stark_queue:{i}` alongside its depth. Queues at 0 appear grey, low depth yellow, high depth red. This is the fastest way to visually confirm even distribution across nodes.

For a single-queue one-shot check:
```powershell
docker exec $(docker ps -qf "name=zk_snark-queue" | Select-Object -First 1) redis-cli LLEN snark_queue:3
```

The selector itself logs its dispatch rate every 5 seconds while running:
```bash
docker service logs zk_verifier-selector --tail 5
# [2026-04-25 22:34:11] Dispatch rate: 312.4 jobs/s (last 5 s)
```

## 5. The Experiment Sequence

The canonical experiment is a four-step flow that produces all the graphs in the paper. Each step feeds a value into the next.

| Step | Purpose | Output |
|---|---|---|
| **A** — VU sweep at weight=0 | Find the system's true throughput ceiling, independent of routing | `KNEE_VU` ≈ 1200 on 80+80 |
| **B** — Weight sweep at moderate VU (400) | Find the cost-weight that maximises composite score (throughput / cost) | `BEST_WEIGHT` |
| **C** — Routing comparison sweep at full VU range | Compare weighted vs round-robin across the full load curve | `comparison_graph.png` |
| **D** — Time-series run at `KNEE_VU` | Long single-load run for the time-series graph | `k6_performance_graph.png` |

Run on the **k6 EC2** in this order. Always set the file descriptor limit before any sweep:
```bash
ulimit -n 65536
```

### Step A — VU sweep at weight=0

```bash
# On k6 EC2: set selector to weight=0 (cost ignored)
curl -X POST "http://<manager-private-ip>:8000/admin/set-weight?snark=0&stark=0"

ulimit -n 65536
python3 sweep_throughput.py \
  --target <manager-private-ip> \
  --vus 50,100,200,400,600,800,1000,1200,1500,2000,3000 \
  --iterations-per-vu 10 \
  --cooldown 15 \
  --stark-ratio 0.0 \
  --output sweep_baseline.csv \
  --clean

python visualize_sweep.py --input sweep_baseline.csv --output sweep_baseline_graph.png
```

Read the **knee** off the throughput curve — the VU level where it stops growing linearly. On 80+80 this is typically ~1200 VUs. Record as `KNEE_VU`.

### Step B — Weight sweep at moderate VU

> ⚠️ **Run this at 400 VUs, not at `KNEE_VU`.** At `KNEE_VU` the system is past cheap-worker saturation, which forces every weight to land at cost=1.5. The weight only has a meaningful effect *below* cheap saturation.

```bash
# On manager: clear Redis state for clean measurement
docker exec $(docker ps -q -f name=zk_proof-queue) redis-cli DEL \
  selector:snark_total_cost selector:snark_total_jobs \
  selector:stark_total_cost selector:stark_total_jobs
docker exec $(docker ps -q -f name=zk_snark-queue) redis-cli FLUSHDB
docker exec $(docker ps -q -f name=zk_stark-queue) redis-cli FLUSHDB
docker exec $(docker ps -q -f name=zk_proof-queue) redis-cli DEL proof_queue
docker service update --force zk_verifier-selector

# On k6 EC2: delete previous CSV (the sweep appends, doesn't overwrite)
rm ~/weight_sweep_results.csv

ulimit -n 65536
python3 weight_sweep.py \
  --target <manager-private-ip> \
  --weights 0,1,2,3,5,7,10,15,20,30,50 \
  --vus 400 \
  --iterations 15000

python visualize_weight_sweep.py
```

Record the weight where the composite score peaks as `BEST_WEIGHT`.

Set the new weight live:
```bash
curl -X POST "http://<manager-private-ip>:8000/admin/set-weight?snark=<BEST_WEIGHT>&stark=1.0"
```

### Step C — Routing comparison

**Run 1 — weighted at `BEST_WEIGHT`:**
```bash
ulimit -n 65536
python3 sweep_throughput.py \
  --target <manager-private-ip> \
  --vus 50,100,200,400,600,800,1000,1200,1500,2000,3000 \
  --iterations-per-vu 10 \
  --cooldown 15 \
  --stark-ratio 0.0 \
  --output sweep_weighted.csv \
  --clean
```

**Switch the selector to round-robin** on the manager:
```bash
docker service update \
  --args "python verifierSelector.py --proof-host proof-queue --proof-port 6379 --snark-host snark-queue --snark-port 6379 --stark-host stark-queue --stark-port 6379 --snark-count 80 --stark-count 80 --routing roundrobin --snark-cost-weight <BEST_WEIGHT> --stark-cost-weight 1.0" \
  zk_verifier-selector

# Confirm Routing=roundrobin from the CURRENT container only (service logs shows old tasks too)
TASK=$(docker service ps zk_verifier-selector --filter "desired-state=running" -q)
docker inspect --format '{{.Status.ContainerStatus.ContainerID}}' $TASK | xargs docker logs 2>&1 | head -3

# Clear Redis state again
docker exec $(docker ps -q -f name=zk_proof-queue) redis-cli DEL \
  selector:snark_total_cost selector:snark_total_jobs \
  selector:stark_total_cost selector:stark_total_jobs
docker exec $(docker ps -q -f name=zk_snark-queue) redis-cli FLUSHDB
docker exec $(docker ps -q -f name=zk_stark-queue) redis-cli FLUSHDB
```

**Run 2 — round-robin:**
```bash
ulimit -n 65536
python3 sweep_throughput.py \
  --target <manager-private-ip> \
  --vus 50,100,200,400,600,800,1000,1200,1500,2000,3000 \
  --iterations-per-vu 10 \
  --cooldown 15 \
  --stark-ratio 0.0 \
  --output sweep_roundrobin.csv \
  --clean
```

**Generate the comparison graph** on your laptop:
```bash
scp -i "zk-authaas-key.pem" ubuntu@<k6-public-ip>:~/sweep_weighted.csv .
scp -i "zk-authaas-key.pem" ubuntu@<k6-public-ip>:~/sweep_roundrobin.csv .

python visualize_comparison.py sweep_weighted.csv sweep_roundrobin.csv \
  --label-a "Weighted (weight=<BEST_WEIGHT>)" \
  --label-b "Round-Robin"
```

The output `comparison_graph.png` has three panels: throughput, p95 latency, and avg cost per job. The shaded region in the cost panel is the **cost saved by weighted routing vs round-robin** — the headline result.

### Step D — Time-series run

```bash
ulimit -n 65536
k6 run \
  -e TARGET=<manager-private-ip> \
  -e VUS=<KNEE_VU> \
  -e ITERATIONS=$((<KNEE_VU> * 20)) \
  -e STARK_RATIO=0.0 \

  --out csv=test_results.csv

# On laptop:
scp -i "zk-authaas-key.pem" ubuntu@<k6-public-ip>:~/test_results.csv .

```

## 7. The v2 Selector

The selector was rewritten in `verifierSelector.py` to remove a single-threaded dispatch bottleneck. Key changes vs v1:

| Concern | v1 (per dispatch) | v2 (per dispatch) |
|---|---|---|
| Weight reads from Redis | 2 round-trips (every job) | 0 — cached in memory, refreshed by background thread every 200 ms |
| Counter updates | 2 separate Redis commands | 1 pipelined call (`incrbyfloat` + `incr`) |
| Total Redis round-trips | **6** | **3** |
| Theoretical ceiling | ~330 dispatches/s | ~650+ dispatches/s |
| Live dispatch logging | None | Logs `Dispatch rate: X.X jobs/s` every 5 s |

Combined with scaling the request-handler to 8 replicas, the system now sustains ~1,800 req/s of real Groth16 verification (≈ 250 actual verifies/s after subtracting status polls).

## 9. Known Architectural Limitations

### Why 80+80 across two nodes (and not 500+500 on a single node)

`snarkjs.groth16.verify` uses `ffjavascript` for BN128 elliptic curve operations. Internally, ffjavascript determines its worker thread count using `os.cpus().length`. Inside a Docker container, `os.cpus()` reports **all host CPUs** (e.g. 96 on a c5.24xlarge) regardless of the container's CPU limit — Docker uses CFS quota-based throttling, not CPU masking.

Result: every SNARK verifier container would spawn **95 worker threads** and use `Atomics.wait()` to synchronise them. With only `0.15` vCPU allocated per container (the limit needed to fit 500 workers on 96 vCPUs), the CFS scheduler throttles the container so aggressively that `Atomics.wait()` blocks indefinitely — the container shows **0% CPU** and the job stays at `"processing"` forever. This is a deadlock, not starvation.

**The fix in this project** — `SNARKVerifierWorker.js` monkey-patches `os.cpus()` at startup to return a single CPU entry before snarkjs/ffjavascript loads:

```javascript
// At top of SNARKVerifierWorker.js — must execute before require("snarkjs")
const os = require("os");
const _realCpus = os.cpus.bind(os);
os.cpus = () => [_realCpus()[0]];
```

This causes ffjavascript to spawn only 1 worker thread per container, making `Atomics` synchronisation viable even at constrained CPU budgets.

**Even with the patch, 500 SNARK workers on one node is infeasible** — with 1 thread per container, each verify needs ~200 ms on 1.0 vCPU. Running 500 workers at 1.0 vCPU requires 500 vCPUs — more than any single EC2 instance provides. The two-node architecture (80 workers × 1.0 vCPU = 80 vCPUs on the dedicated SNARK node) is the practical ceiling for real Groth16 on a single c5.24xlarge.

**A second hard limit:** at ~1000 containers on a single node, the Docker overlay VXLAN forwarding table is overwhelmed and service VIPs (used by selector → worker queues) start flapping with `Error 113: No route to host`. Splitting the pools across two nodes keeps each node well below this limit.

**Academic justification** — in a production SaaS deployment, Groth16 verifiers would be compiled Rust binaries (e.g. `arkworks`, `bellman`) completing in <10 ms per proof. The bottleneck demonstrated here is specific to Node.js + WASM. The 80+80 configuration validates the routing, queueing, and cost-weighted scheduling architecture — which is the system under study — at meaningful concurrency with real cryptographic work.

**Scaling beyond 80+80** with real Groth16 is straightforward: add more worker EC2 nodes to the Swarm. Each c5.24xlarge worker hosts another 80 SNARK workers, giving linear horizontal scaling. 6 worker nodes ≈ 500 SNARK workers — the intended production topology for a SaaS ZK verification service.

## 11. Troubleshooting

**Manager and worker EC2s can't ping each other / `docker swarm join` times out.**
The two backend nodes must share a single security group with a self-referencing "All traffic" inbound rule. Separate SGs block intra-cluster traffic. See Step 1 of the checklist.

**SNARK jobs stuck at `"processing"`, verifier shows 0% CPU.**
`Atomics.wait()` deadlock from missing `os.cpus()` monkey-patch. Confirm the patch is at the top of `SNARKVerifierWorker.js` (before `require("snarkjs")`) and that the SNARK image was rebuilt after the patch was added.

**SNARK service stuck at `0/80` on the worker node.**
Image not built on that node. Run `docker build -f Dockerfile.snark -t zk-authaas/snark-verifier:latest .` on the worker EC2.

**SNARK tasks scheduled on the manager (or STARK on the worker).**
Node labels missing. On the manager: `docker node update --label-add pool=stark $(docker node ls -q --filter role=manager)` and `docker node update --label-add pool=snark $(docker node ls -q --filter role=worker)`.

**Cost line stays flat at 1.5 across all weights in the weight sweep.**
The sweep is running above cheap-worker saturation. Re-run at a lower VU (typically 400, not `KNEE_VU`).

**Cost line stays flat at 1.0 across all weights.**
The opposite — load is far below cheap capacity, so even tiny weights route everything to cheap. Either push more VUs or reduce the cheap pool size to make saturation reachable.

**k6 freezes or is killed mid-sweep.**
Two causes: (1) file descriptor limit — run `ulimit -n 65536` in the same shell before every sweep; (2) OOM — each VU uses ~1–2 MB of RAM; the default `c5.2xlarge` (16 GiB) handles up to ~5,000 VUs. Upgrade to `c5.4xlarge` if you need more.

**Weight sweep graph is spiky, multiple data points per weight.**
The CSV from a previous run wasn't deleted — `weight_sweep.py` appends rather than overwrites. Run `rm ~/weight_sweep_results.csv` before each sweep.

**Selector logs show old `ConnectionError` or stale dispatch rates after `service update`.**
`docker service logs` shows output from all historical task instances, including dead containers. To read only the currently running container:
```bash
TASK=$(docker service ps zk_verifier-selector --filter "desired-state=running" -q)
docker inspect --format '{{.Status.ContainerStatus.ContainerID}}' $TASK | xargs docker logs 2>&1 | head -5
```

**Cost panel in `comparison_graph.png` shows the same value for both algorithms.**
The selector wasn't actually switched between runs. Confirm with the current-container log command above — the startup line must show `Routing=weighted` for run 1 and `Routing=roundrobin` for run 2.

**`docker service scale` stalls, no errors.**
Kernel inotify limit exhausted (`max_user_instances=128`). On both backend nodes:
```bash
sudo sysctl fs.inotify.max_user_instances=8192
sudo sysctl fs.inotify.max_user_watches=524288
```

**`docker stack rm` leaves networks behind.**
Wait ~15 seconds before redeploying. Swarm tears down overlay networks asynchronously. If it hangs indefinitely: `docker swarm leave --force && docker swarm init`.

**`watch: command not found` on Windows.**
That's a Linux command. Use `monitor_queues.ps1` instead.
