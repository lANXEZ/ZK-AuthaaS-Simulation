# ZK-AuthaaS Simulation

A distributed zero-knowledge proof verification system built to study load-balancing behaviour across heterogeneous verifier pools. The system is composed of a FastAPI request handler, a custom verifier selector that routes proofs by scheme and weighted cost, and pools of SNARK and STARK verifier workers connected through Redis queues. Load is driven by k6.

SNARK verification uses **real Groth16 proof verification** via `snarkjs` running inside each worker container. STARK verification uses a timing placeholder. Load is generated by k6 — either from a local machine for development, or from a dedicated k6 EC2 instance in the same AWS VPC for accurate cloud benchmarking.

This project is deployed with **Docker Swarm** so that verifier counts can be scaled with a single command from 10 up to 1000 replicas without editing the compose file.

---

## 1. Prerequisites

### For local development (your laptop)

- [Docker Desktop](https://www.docker.com/products/docker-desktop/) (Windows / macOS) or Docker Engine (Linux) — Swarm mode is included
- [k6](https://k6.io/docs/getting-started/installation/) for load generation
- [Python 3.8+](https://www.python.org/) for the sweep and visualization scripts
- Python packages: `pip install pandas matplotlib numpy`

> **Note:** Node.js is **not** required on your local machine. The SNARK worker runs inside a `node:18-slim` Docker container — the build handles everything.

On Windows PowerShell the monitoring script also requires loosening the execution policy once:
```powershell
Set-ExecutionPolicy -Scope CurrentUser RemoteSigned
```

### For AWS deployment

Two EC2 instances are required (see [Section 9](#9-deploying-on-aws)):

| Instance | Role | Suggested type |
|---|---|---|
| `zk-authaas-backend` | Runs the Docker Swarm stack | `c5.4xlarge` (50+50 verifiers) |
| `zk-authaas-k6` | Runs k6 load generator | `t3.small` |

Both instances must be in the **same VPC and Availability Zone**. k6 fires at the backend over the private network, keeping network overhead below 1 ms and out of the measurements.

## 2. Architecture

```
 ┌─────────┐   POST /verify/submit   ┌─────────────────┐
 │   k6    │ ──────────────────────► │ request-handler │
 └─────────┘                         │   (FastAPI)     │
      ▲                              └────────┬────────┘
      │ poll /verify/status                   │ lpush proof_queue
      │                                       ▼
      │                              ┌─────────────────┐
      │                              │   proof-queue   │  (Redis :6379)
      │                              └────────┬────────┘
      │                                       │ brpop proof_queue
      │                                       ▼
      │                              ┌─────────────────┐
      │                              │ verifier-       │  cost-weighted least-queue
      │                              │   selector      │  + pub/sub feedback
      │                              └────────┬────────┘
      │                         lpush snark_queue:{i}   lpush stark_queue:{i}
      │                                       ▼                     ▼
      │                              ┌───────────────┐   ┌───────────────┐
      │                              │  snark-queue  │   │  stark-queue  │
      │                              │ (Redis :6380) │   │ (Redis :6381) │
      │                              └───────┬───────┘   └───────┬───────┘
      │          brpop snark_queue:{i}       │                   │  brpop stark_queue:{i}
      │                                      ▼                   ▼
      │                              ┌──────────────┐   ┌──────────────┐
      │                              │snark-verifier│   │stark-verifier│
      │                              │ (N replicas) │   │ (N replicas) │
      │                              │  snarkjs     │   │  placeholder │
      │                              │ Groth16 real │   │  timing only │
      │                              └──────┬───────┘   └──────┬───────┘
      │                                     │                  │
      │                                     └────── publish ───┴── verifier_feedback
      │                                                        │
      └──────── set status:{job_id} ───────────────────────────┘
```

A single Redis per verifier type hosts one list per worker node (`snark_queue:{index}` / `stark_queue:{index}`). Each worker reads only its own key, giving the selector fine-grained routing control while keeping the total number of Redis processes small (3 regardless of replica count).

Each SNARK worker loads `verification_key.json` once at startup and calls `snarkjs.groth16.verify(vKey, publicSignals, proof)` on every job. The measured latency therefore reflects real BN128 pairing computation, not a sleep.

## 3. Build and Deploy (local)

Docker Swarm handles orchestration. These same commands work on your laptop and on the backend EC2.

```bash
docker swarm init
docker compose build                           # build all service images (~3-5 min first run due to snarkjs install)
docker stack deploy -c docker-compose.yml zk   # deploy the stack
docker service ls                              # wait until all services reach target replicas
```

Expected output of `docker service ls` at default scale (10+10):

```
NAME                      MODE         REPLICAS   IMAGE
zk_proof-queue            replicated   1/1        redis:latest
zk_snark-queue            replicated   1/1        redis:latest
zk_stark-queue            replicated   1/1        redis:latest
zk_request-handler        replicated   1/1        zk-authaas/request-handler:latest
zk_verifier-selector      replicated   1/1        zk-authaas/verifier-selector:latest
zk_snark-verifier         replicated   10/10      zk-authaas/snark-verifier:latest
zk_stark-verifier         replicated   10/10      zk-authaas/stark-verifier:latest
```

### Scaling on the fly

```bash
docker service scale zk_snark-verifier=50 zk_stark-verifier=50
```

> **Important:** the selector is started with `--snark-count` and `--stark-count` arguments that must match the replica counts. If you scale on the fly, also update and redeploy the selector, or edit `docker-compose.yml` and run `docker stack deploy` again.

## 4. Run the Load Test

### Local (laptop → localhost)

```bash
k6 run load_test.js
```

Defaults target `localhost:8000` with 50 VUs and 500 iterations — suitable for a local 10+10 deployment.

### AWS (k6 EC2 → backend EC2 private IP)

SSH into the k6 EC2 and run:

```bash
k6 run \
  -e TARGET=<backend-private-ip> \
  -e VUS=200 \
  -e ITERATIONS=5000 \
  -e STARK_RATIO=0.0 \
  load_test.js \
  --out csv=test_results.csv
```

Always use the **private IP** of the backend EC2, not the public IP. Private IP traffic stays on the AWS internal network and adds less than 1 ms RTT, keeping network overhead out of the `snark_verification_time` measurements.

### Configuration reference

All parameters are environment variables — no need to edit the file:

```bash
k6 run -e TARGET=localhost   load_test.js          # local (default)
k6 run -e STARK_RATIO=0.0    load_test.js          # pure SNARK
k6 run -e STARK_RATIO=1.0    load_test.js          # pure STARK
k6 run -e STARK_RATIO=0.5    load_test.js          # 50/50 mix (default)
```

### Suggested load profiles

| Deployment | Verifiers | VUS | ITERATIONS |
|---|---|---|---|
| Local laptop | 10 + 10 | 50 | 500 |
| AWS `c5.4xlarge` | 50 + 50 | 200 | 5 000 |
| AWS `c5.24xlarge` (full run) | 500 + 500 | 1 000 | 50 000 |

## 5. Monitor the Queues Live

Use the included PowerShell script to watch all queues in real time with colour-coded depth:

```powershell
.\monitor_queues.ps1
.\monitor_queues.ps1 -SnarkCount 50 -StarkCount 50 -Refresh 2
```

It displays the incoming `proof_queue` and every `snark_queue:{i}` / `stark_queue:{i}` alongside its depth. Queues at 0 appear grey, low depth yellow, high depth red. This is the fastest way to visually confirm even distribution across nodes.

For a single-queue one-shot check:

```powershell
docker exec $(docker ps -qf "name=zk_snark-queue" | Select-Object -First 1) redis-cli LLEN snark_queue:3
```

## 6. Find Maximum Throughput (Sweep)

`sweep_throughput.py` runs k6 at a series of VU levels and records per-run metrics to `sweep_results.csv`:

```bash
python sweep_throughput.py                          # default local sweep, all SNARK
python sweep_throughput.py --stark-ratio 0.5        # 50/50 mix
python sweep_throughput.py --vus 50,100,200,400,800 --clean
python sweep_throughput.py --target 172.31.45.12    # target a remote EC2
```

The "knee" in the resulting curve — the VU level where `throughput_req_per_sec` stops growing — is the optimal operating point.

```bash
python visualize_sweep.py   # produces sweep_throughput_graph.png
```

## 9. Deploying on AWS

### Account access

This project uses a **paid-tier AWS account** (borrowed from a professor). Do not use AWS Educate or free-tier accounts — service limits and network restrictions on those tiers interfere with Docker Swarm overlay networking and spot instance availability.

Contact the account holder before starting any session and confirm the current billing balance. Set a **budget alert at $20** in AWS Billing before launching anything.

### Two-instance layout

```
Your laptop  ──SSH──►  zk-authaas-backend (c5.4xlarge)   ← Docker Swarm stack
                               │  AWS private network (same VPC + AZ)
             ──SSH──►  zk-authaas-k6 (t3.small)  ──HTTP:8000──►  backend private IP
```

### Step 1 — IAM and SSH (one-time per team)

- Log in as an IAM user with `AmazonEC2FullAccess` (do not use the root account)
- Go to **EC2 → Key Pairs → Create key pair**, name it `zk-authaas-key`, download the `.pem` file
- `chmod 400 zk-authaas-key.pem`
- Pick one region and use it for every resource in this project (`us-east-1` is a good default)

### Step 2 — Launch the backend EC2

1. **EC2 → Launch Instance**
2. Name: `zk-authaas-backend`
3. AMI: **Ubuntu 22.04 LTS**
4. Instance type: `c5.4xlarge` (16 vCPU, 32 GB RAM)
5. Key pair: `zk-authaas-key`
6. Security group — create `zk-authaas-backend-sg` with:
   - SSH (port 22) from **My IP**
   - Custom TCP (port 8000) from **My IP** *(for manual curl checks)*
7. Storage: 30 GB gp3
8. Launch — record the **Public IPv4** and the **Private IPv4**

### Step 3 — Launch the k6 EC2

1. **EC2 → Launch Instance**
2. Name: `zk-authaas-k6`
3. AMI: **Ubuntu 22.04 LTS**
4. Instance type: `t3.small` (2 vCPU, 2 GB RAM)
5. Key pair: `zk-authaas-key`
6. Network — **critical**: expand and set:
   - VPC: **same VPC** as the backend
   - Subnet: **same subnet / AZ** as the backend (check the backend's AZ in the EC2 console and match it exactly)
7. Security group — create `zk-authaas-k6-sg`:
   - SSH (port 22) from **My IP**
   - *(No inbound rules needed — k6 only makes outbound calls)*
8. Storage: 8 GB gp3
9. Launch — record the **Private IPv4**

### Step 4 — Allow k6 to reach the backend

1. **EC2 → Security Groups → `zk-authaas-backend-sg` → Inbound rules → Edit**
2. Add rule: Custom TCP · Port 8000 · Source: `<k6-private-ip>/32`
3. Save

### Step 5 — Set up the backend EC2

```bash
ssh -i zk-authaas-key.pem ubuntu@<backend-public-ip>

# Install Docker
sudo apt update && sudo apt install -y docker.io git
sudo systemctl enable --now docker
sudo usermod -aG docker ubuntu
exit   # log out so the group change takes effect

ssh -i zk-authaas-key.pem ubuntu@<backend-public-ip>
docker ps   # confirm no sudo needed

# Get the code
git clone <your-repo-url> zk-authaas
cd zk-authaas

# Init Swarm, build images, deploy
docker swarm init
docker compose build   # first build takes ~3-5 min (snarkjs npm install)
docker stack deploy -c docker-compose.yml zk
docker service ls      # wait ~30s for all replicas to reach target
```

Quick sanity check once `zk_request-handler` shows `1/1`:

```bash
curl -s -X POST http://localhost:8000/verify/submit \
  -H "Content-Type: application/json" \
  -d '{"scheme":"stark","proof":"x","public_inputs":["a"]}' | python3 -m json.tool
# Expected: {"status": "accepted", "job_id": "..."}
```

### Step 6 — Set up the k6 EC2

Open a second terminal on your laptop:

```bash
ssh -i zk-authaas-key.pem ubuntu@<k6-public-ip>

# Install k6
sudo apt update && sudo apt install -y gpg curl
curl -s https://dl.k6.io/key.gpg | sudo gpg --dearmor -o /usr/share/keyrings/k6-archive-keyring.gpg
echo "deb [signed-by=/usr/share/keyrings/k6-archive-keyring.gpg] https://dl.k6.io/deb stable main" \
  | sudo tee /etc/apt/sources.list.d/k6.list
sudo apt update && sudo apt install -y k6
k6 version
```

Copy the load test script from your laptop (run this command on your laptop, not the EC2):

```bash
scp -i zk-authaas-key.pem \
  "load_test.js" \
  ubuntu@<k6-public-ip>:~/load_test.js
```

### Step 7 — Run the test

On the k6 EC2, using the backend's **private IP**:

```bash
k6 run \
  -e TARGET=<backend-private-ip> \
  -e VUS=200 \
  -e ITERATIONS=5000 \
  -e STARK_RATIO=0.0 \
  load_test.js \
  --out csv=test_results.csv
```

### Step 8 — Collect results

Copy results from the k6 EC2 back to your laptop:

```bash
# Run on your laptop
scp -i zk-authaas-key.pem ubuntu@<k6-public-ip>:~/test_results.csv ./test_results.csv
python visualize_k6.py
```

### Step 9 — TEAR DOWN (every session, no exceptions)

On the backend EC2:

```bash
docker stack rm zk
exit
```

Then in the AWS Console:

1. **EC2 → Instances** — select both `zk-authaas-backend` and `zk-authaas-k6`
2. **Instance State → Terminate**
3. Wait for both to show `terminated`
4. Confirm the EC2 dashboard shows **0 running instances** before closing the tab

A running `c5.4xlarge` on-demand costs roughly **$0.68/hr**. A forgotten instance left running for a week is a ~$115 charge. Always terminate, never stop.

## 11. Troubleshooting

**Services stuck in `0/N` or restarting.**
Check the failure reason: `docker service ps zk_snark-verifier --no-trunc`. Most common causes are memory overcommit (reduce replicas or raise the memory limit in `docker-compose.yml`) or a failed image build.

**SNARK workers show `Error: Invalid proof` in logs.**
The `verification_key.json` inside the image does not match the proof being sent. Confirm the file was copied to the project root before running `docker compose build`, then rebuild with `docker compose build --no-cache zk_snark-verifier`.

**SNARK workers are slower than expected / high CPU.**
Normal — `snarkjs.groth16.verify()` performs BN128 elliptic curve pairings which are genuinely compute-intensive. If replicas are saturated, scale up: `docker service scale zk_snark-verifier=50`.

**k6 reports many `submit_failures`.**
The request-handler is saturated. Scale it: `docker service scale zk_request-handler=3`.

**k6 reports correct job submissions but `failed_verifications` are high.**
The SNARK workers are processing jobs but returning `false`. This usually means the proof or public signals in `load_test.js` were modified and no longer match the verification key. Restore `SNARK_PROOF` and `SNARK_PUBLIC_SIGNALS` from `zk-authaas-snark/Test/`.

**Queues stay full and never drain.**
Either no worker replicas are running (`docker service ls`) or the selector's `--snark-count` / `--stark-count` don't match the actual replica count, so some queue keys receive jobs that no worker is listening on.

**k6 EC2 gets `connection refused` on port 8000.**
Either the backend security group is missing the rule for the k6 EC2's private IP (Section 9 Step 4), or you used the backend's public IP instead of the private IP in the `TARGET` variable.

**`docker stack rm` leaves networks behind.**
Wait ~15 seconds before redeploying. Swarm tears down overlay networks asynchronously.

**`command not found: watch` on Windows.**
That is a Linux command. Use `monitor_queues.ps1` instead.
